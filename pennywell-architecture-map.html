Pennywell Scholarship Operating System

How the Airtable base works today, why the dashboard shows zero, and the wiring plan that turns it into one source of truth feeding the Scholarship Wallet.

Owner: Chris — Chief Automation ArchitectPhase: 1 — Build the Operating SystemStatus: Audit complete
The one-line summary: the base already has the right pieces, but nothing is wired together. 249 real applications sit orphaned, and the dashboard reads a manual table almost nobody fills, so it reports zero. The job is mostly connection and consolidation, not new construction. Only two pieces are genuinely new builds: the matching engine and the client portal. A one-client pilot gets us to a working dashboard by the Week 3 target.

  • Applications is orphaned. 249 records with rich workflow, but 0/249 linked to a Student or Scholarship and only 28/249 to a Client.
  • Activity is split across three places. Applications (249), the Nathalia one-off table (59), and hand-written Weekly Reports (6). No single source of truth.
  • The dashboard reads the wrong source. It pulls only from manual Weekly Reports, which stopped being filled around April, so it shows zero.
  • Clients and Student Profiles duplicate each other. The same academic fields (GPA, SAT, major, race) live on both tables.
  • Intake is misfiring. The Sheets-to-Clients automation is off, while four email automations fire on every create and update. A duplicate client and two blank ghost rows already slipped in.
  • One thing already works: Hub matching. 1,678 of 1,922 scholarships are matched to clients.

3 The core problem

The problem isn't missing features. The team has done real work — hundreds of applications. The problem is that the tables aren't connected, the work is recorded in several parallel places, and the dashboard looks at a manual table instead of the live data. So the system can't show what's actually happening, which reads as poor performance to clients even when the work is strong.

4 The target architecture

Five core tables wired into one source of truth that feeds the client-facing Scholarship Wallet. Matching and the portal are the only new builds.

6 The work: fix, connect, build

Most of this is connection and cleanup. New construction is the small part.

Consolidate

Make Applications the single source. Adopt the strategist's tracker format. Split Clients vs Students cleanly. Fold the Nathalia table in. Retire the empty Tasks table.

Clean

Merge the duplicate client, delete the two ghost rows, replace the record-ID "unique key" with an email-based key, and populate the empty Service Tier field.

Connect

Link every Application to its Client, Student, and Scholarship (today: zero linked). Build rollups so all totals compute automatically.

Build (new)

The matching engine (criteria-based per-client views) and the client-facing Scholarship Wallet. These two are the only genuinely new pieces. Matching engine must support a per-client manual override: Sienna can remove a wrongly-matched scholarship from one client's list without deleting it from the Hub (it stays available to everyone else). Matching criteria include an education-level tag (high school senior / undergraduate / graduate) so scholarships are matched to the student's stage — added at Sienna's request (6/21 email).

8 Decisions needed from the team

Five things only leadership can settle. Each one either blocks a piece of the build or changes how it gets built.

  • 1. Airtable plan tier hard blocker
    The free plan caps a base at 1,000 records. The Hub already holds 1,922, and we'll add hundreds of application rows per client, so the free plan physically cannot hold the data. Nothing else works until this is settled.
    Ask: are we on a paid Airtable plan with room for 50,000+ records?
  • 2. Build a new clean base, migrate into it confirmation
    Agreed in the 6/19 meeting: instead of untangling the current base (249 orphaned applications, automations firing on every edit, ghost rows, a duplicate client), we build a fresh base inside the same workspace and copy clean data in. The old base stays as a read-only backup until the new one is validated, then it's archived. No extra Airtable cost — bases are unlimited on any plan; you pay per editor seat, not per base, and the new base gets its own 50,000-record cap. Migration runs in dependency order so links resolve: Clients → Students → Scholarships + Hub → Applications → Awards → Emails (clean as you go: dedupe the duplicate client, drop the two ghost rows, fold Nathalia into Applications, skip the empty Tasks table, swap the record-ID key for an email key). The stale Weekly Reports table isn't migrated — auto-rollups replace it.
    Confirm: cleared to build a new base and migrate, keeping the old one as backup until sign-off?
  • 3. Client portal tool and budget
    The Scholarship Wallet lets each family log in and see only their own data. Airtable can't do that cheaply or securely on its own (no true per-client row-level security; external access is billed per seat). The fix is a front-end tool that sits on top of the Airtable base and adds logins, branding, and row-level security. Two real options — both prototype for free, both are add-ons to the Airtable Team plan, not replacements for it:
     Softr (Professional)Noloco (Pro)
    Price (annual)~$139/mo~$99/mo
    Client logins included10050 (then $1 each)

    Both do proper row-level security and branding. At 8–50 clients Noloco is the cheaper fit; past ~100 clients Softr's seat math pulls ahead. Note: "NocoDB" (seen in the meeting notes) is a different product — an open-source Airtable replacement, not a portal builder — so it's not in this comparison.
    Ask: Softr or Noloco for the portal? Budget is $0 to prototype, then ~$99–139/mo at launch.
  • 4. Service tiers and sensitive data
    The Service Tier field is blank for every client, and we need each one's tier (Done-For-You vs Intel) because it drives what they get and how they're reported. The base also holds minors' personal data (date of birth, race, income, FAFSA), so we need rules for who sees what, especially once clients log into the portal.
    Ask: what tier is each active client, and what are the data-handling rules?
  • 5. Strategist workflow confirmation
    The system only works if Sienna logs her applications in Airtable, not personal Excel files. The Operating Manual already mandates this; it just has to hold in practice, or the dashboard shows empty data like it does today.
    Ask: we're committed to Airtable-only logging, and I'll train Sienna on it, right?

In short: #1 is the hard blocker, #2 is the build approach (agreed — just needs sign-off), #3 shapes the build, #4 fills in details you'll need, and #5 is the behavior that keeps the whole thing alive.
