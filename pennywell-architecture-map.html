Phase 1 · Week 1 Deliverable — Airtable Architecture Map

Pennywell Scholarship Operating System

How the Airtable base works today, why the dashboard shows zero, and the wiring plan that turns it into one source of truth feeding the Scholarship Wallet.

Owner: Chris — Chief Automation ArchitectPhase: 1 — Build the Operating SystemStatus: Audit complete · updated after June 19 meeting
The one-line summary: the base already has the right pieces, but nothing is wired together. 249 real applications sit orphaned, and the dashboard reads a manual table almost nobody fills, so it reports zero. The plan agreed June 19 is to build a fresh, clean Airtable base and migrate the good data into it, wired correctly from the start, rather than untangling the old one in place. Only two pieces are genuinely new builds: the matching engine and the client portal. A one-client pilot gets us to a working dashboard inside the 4-week build-out.

  • Applications is orphaned. 249 records with rich workflow, but 0/249 linked to a Student or Scholarship and only 28/249 to a Client.
  • Activity is split across three places. Applications (249), the Nathalia one-off table (59), and hand-written Weekly Reports (6). No single source of truth.
  • The dashboard reads the wrong source. It pulls only from manual Weekly Reports, which stopped being filled around April, so it shows zero.
  • Clients and Student Profiles duplicate each other. The same academic fields (GPA, SAT, major, race) live on both tables.
  • Intake is misfiring. The Sheets-to-Clients automation is off, while four email automations fire on every create and update. A duplicate client and two blank ghost rows already slipped in.
  • One thing already works: Hub matching. 1,678 of 1,922 scholarships are matched to clients.

3 The core problem

The problem isn't missing features. The team has done real work — hundreds of applications. The problem is that the tables aren't connected, the work is recorded in several parallel places, and the dashboard looks at a manual table instead of the live data. So the system can't show what's actually happening, which reads as poor performance to clients even when the work is strong.

4 The target architecture

Five core tables wired into one source of truth that feeds the client-facing Scholarship Wallet. Matching and the portal are the only new builds.

6 The work: fix, connect, build

A clean new base, with the good data migrated in and wired correctly. New construction is the small part.

Build new + migrate

Stand up a fresh, clean base with the target tables, then migrate the good data in. Applications becomes the single source; Clients and Students split cleanly; the Nathalia table and the strategists' Excel fold into Applications. The old messy base is left behind, not edited.

Clean by migration

Because we migrate into a new base, bad data simply isn't carried over: the duplicate client, the ghost rows, and the broken record-ID key all stay behind. Each scholarship gets a unique ID for cross-cycle dedup, and Service Tier is filled in per client.

Connect

Link every Application to its Client, Student, and Scholarship (today: zero linked). Build rollups so all totals compute automatically.

Build (new)

The matching engine and the client-facing Scholarship Wallet (via Softr/Noloco), plus a native Airtable intake form, an Essay Hub for reusing essays, and per-client removal of ineligible scholarships (kept in the Hub). These are the genuinely new pieces.

8 Decisions: settled and still open

The June 19 team meeting resolved most of the open questions. Here is where things stand.

Settled on June 19:
  • Airtable Team plan confirmed and being upgraded (clears the record-cap blocker).
  • New clean base, migrate into it rather than fixing the old one in place.
  • Softr or Noloco for the client portal, to avoid Airtable's pricey external sharing.
  • Native Airtable forms replace the Google Form for intake.
  • Essay Hub table, unique scholarship IDs, and per-client removal of ineligible scholarships added to scope.
  • Airtable-only workflow (Excel discontinued) and restricted per-client dashboard access.
  • 4-week Phase 1 build-out, and Sienna to send all client Excel files for migration.

Still open, to raise Monday:

  • Capacity flag: the stated goal of a 75,000-scholarship Hub exceeds the Team plan's 50,000-record cap. Team is fine for now, but 75k will need the Business plan (125k).
  • Junk-mail filter criteria for the email automation (flagged for a later session).
  • Per-client Service Tier values to fill in, and the rules for handling sensitive student data in the portal.
